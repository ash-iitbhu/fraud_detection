## Project Overview

End-to-end unsupervised fraud detection pipeline for heterogeneous financial transaction logs.
Designed to be production-realistic: all features are point-in-time correct, models are
independently trained, and the evaluation strategy is honest about the absence of ground-truth labels.

## Parsing Logic

The parser uses a **4-layer cascade**, trying each in order until a row is successfully parsed:

| Layer | Method | Handles |
|---|---|---|
| 1 | **Regex Cascade** | 8 compiled patterns (F1–F8) covering all known log formats |
| 2 | **Heuristic Fallback** | Field-by-field micro-pattern extraction when full regex fails |
| 3 | **spaCy EntityRuler** | Rule-based NLP patterns (NOT statistical NER — purely deterministic) |
| 4 | **LLM Fallback** | Optional API call (env-gated — requires `ANTHROPIC_API_KEY`) |
| 5 | **PARSE_FAILURE** | Unparseable rows preserved with `parse_success=False` |

**Key parsing decisions:**
- `None` / `null` values preserved as `UNKNOWN` string — not Python `None` — with boolean flags `is_city_missing`, `is_device_missing`
- Mixed currency positions (`€4821.85` vs `951.85$`) handled by dual-capture regex group
- Mixed date formats (`YYYY-MM-DD` and `DD/MM/YYYY`) handled by `dateutil` with `dayfirst` detection
- Transaction type synonyms normalized to 4 canonical types: `deposit`, `withdrawal`, `purchase`, `transfer`
- MALFORMED rows counted and preserved — their temporal clustering is itself a fraud signal

**Parse rate achieved: 92.6%** (all remaining 7.4% are genuine MALFORMED_LOG entries)

## Modeling Approach

### Three-Layer Detection Stack

```
Transaction
    │
    ▼
[Layer 1] Rule Engine ──────────────────────────► rule_flag_count + rule_explanations
    │         9 user-context-aware rules
    │
    ▼
[Layer 2] Isolation Forest ─────────────────────► if_score (continuous anomaly score)
    │         Continuous + rarity-encoded features
    │         Global outlier detection
    │
    ▼
[Layer 3] Autoencoder + Entity Embeddings ───────► ae_score (reconstruction error)
    │         user_id/city/device/txn_type embeddings
    │         Multivariate pattern anomaly detection
    │
    ▼
[Ensemble] Weighted composite score ─────────────► final_risk_score → TIER 1/2/3/NORMAL
```

### Why These Three Are Non-Redundant

| Model | What It Detects | Categorical Handling | Output |
|---|---|---|---|
| Rule Engine | Known fraud patterns (velocity, new city + high amount, late night) | Explicit logic | Binary flags |
| Isolation Forest | Global population outliers — transactions rare in continuous feature space | Rarity encodings (scalar) | Continuous score |
| Autoencoder | Multivariate pattern breaks — combinations of features that never co-occur in normal behavior | Entity embeddings (16-dim learned representations) | Reconstruction error |

**Why categoricals are handled differently per model:**
- IF: Uses rarity scores (`city_global_rarity`, `device_global_rarity`, `device_city_pair_rarity`) — scalar, directly interpretable, no dimensionality issue
- AE: Uses learned embedding layers — 500 users → 16-dim behavioral fingerprint. Users with similar behavior get similar embeddings. Anomaly = user behaves unlike their learned fingerprint.
- **This preserves model independence** — the AE is not trained on IF outputs. Both models see categorical signals through different, independently computed lenses.

## Feature Engineering

All 25+ features are **point-in-time correct**:
- Sorted by `(user_id, timestamp)` before any aggregate
- `.shift(1)` applied before every `.expanding()` window
- This makes features equivalent to real-time inference — no future data used

**Cold-start handling:** Users with < 3 prior transactions receive global population priors (median amount, global std) rather than NaN, with an explicit `is_cold_start` flag.

| Feature Group | Examples |
|---|---|
| Amount | `amount_zscore`, `amount_global_percentile`, `is_round_amount` |
| Temporal | `hour_of_day`, `is_late_night`, `time_since_last_txn_sec`, `user_txn_velocity_1h` |
| Location | `is_new_city`, `user_city_freq`, `city_global_rarity`, `user_location_entropy` |
| Device | `is_new_device`, `device_global_rarity`, `device_city_pair_rarity` |
| User Behavior | `user_avg_amount_hist`, `user_late_night_ratio`, `user_frequency_hist` |
| RFM | `user_recency_days`, `user_monetary_hist` |
| Metadata | `missing_field_count`, `is_cold_start`, `txn_type_encoded` |

## Evaluation Strategy (No Ground-Truth Labels)

1. **Synthetic Injection Test** — 100 known fraud patterns injected at data generation. Measures recall in top-K flagged transactions.
2. **Precision-at-K** — Of the top 20/50/100 by risk score, what fraction are genuine anomalies? This is the operational metric used by fraud analyst teams.
3. **Cross-Model Agreement Matrix** — Pairwise overlap in top-5% flags. Target: < 50% overlap (non-redundant detectors).
4. **Score Distribution Analysis** — Bimodal distribution indicates the model found meaningful structure.
5. **Anomaly Rate Check** — Flagged rate should align with industry baseline (0.5–3% for fintech).

## Findings & Business Impact

### Risk-Response Framework

| Tier | Threshold | Action |
|---|---|---|
| TIER 1 | ≥ 0.75 | Auto-block + immediate notification + fraud team escalation |
| TIER 2 | 0.50–0.75 | Step-up authentication (OTP / biometric) |
| TIER 3 | 0.25–0.50 | Silent monitoring, enhanced velocity tracking |
| NORMAL | < 0.25 | No action |

### Observed Results (on synthetic dataset)
- Parse rate: **92.6%** (100% of genuine logs parsed, 7.4% = genuine MALFORMED entries)
- Tier 1 flagged: **~1.8%** of transactions (within 0.5–3% industry benchmark)
- SHAP analysis shows `amount_zscore`, `is_late_night`, and `city_global_rarity` as top anomaly drivers

### Estimated Business Value
At 1,000 transactions/day, 1% genuine fraud rate, 80% Tier 1 precision:
- **~8 genuine fraud cases auto-blocked per day**
- At avg fraud value of £1,500 → **£12,000/day in prevented losses**
- Annual exposure reduction: **~£4.4M** (before operational cost of false positives)

## Production Architecture (Interview Reference)

```
[Transaction Event]
        │
        ▼
[Feature Store] ──── per-user rolling aggregates (Redis / Feast)
        │
        ▼
[Model Scoring Service]
    ├── Rule Engine     (stateless, < 1ms)
    ├── Isolation Forest (pre-trained, < 5ms)
    └── Autoencoder     (pre-trained, < 10ms)
        │
        ▼
[Ensemble Scorer] → risk_score + tier
        │
        ▼
[Action Router]
    ├── TIER 1 → Block API + Alert
    ├── TIER 2 → Step-up Auth API
    └── TIER 3 → Monitoring DB
        │
        ▼
[Analyst Review Queue] → feedback → weight tuning → retraining
```

**Batch retraining cadence:** Weekly on rolling 90-day window
**Model drift monitoring:** Alert if flagged rate exceeds 2× rolling baseline
**Cold-start mitigation:** Segment priors + graph identity linkage (device fingerprint matching)
